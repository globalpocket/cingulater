# Cingulater: Autonomous Agent Engine

Cingulater は、LLM（大規模言語モデル）の推論ストリームをリアルタイムで監視・介入・修正するパイプラインを備えた、自律型エージェント実行エンジンです。

単にユーザーのプロンプトを LLM に渡し、その結果を返すだけのシステムではありません。Cingulater の真の価値は、LLM の出力の不確実性（ハルシネーションや行動の欠如）をシステム層で吸収し、タスクの完遂を自動的に担保する **Interceptor（インターセプター）アーキテクチャ** にあります。

## 🧠 Core Autonomous Mechanisms

Cingulater が自律的である根拠は、以下の実装ロジックによってシステムが独自の判断と修復を行う点にあります。

### 1. Reflection & Action Completion (内省と行動の自動補完)

LLM はしばしば「テキストで回答して満足し、実際のツール（関数）を実行しない」という問題を抱えます。`ReflectionInterceptor` はこの問題を自律的に解決します。

* **検知**: LLM の出力ストリームを監視し、`ToolCall` が一度も発生せずにワークフローが終了した場合に介入します。
* **意図抽出 (Intent Extraction)**: 生成された回答の末尾から、ユーザーの真の意図を短い英語フレーズとして抽出します。
* **セマンティック検索**: 抽出した意図と利用可能な全ツールのドキュメントを比較します。この際、MCP (Model Context Protocol) 経由で外部の Reranker (`mcp-reranker`) を呼び出し、最も適合するツールをスコアリングして決定します。
* **強制実行**: 最適なツールが見つかると、システムが LLM に代わって `SystemToolCallEvent` を発火させ、回答内容を引数にマッピングしてツールを自動実行します。

### 2. Self-Healing from Hallucinations (ハルシネーションの自己修復)

LLM が「存在しないツール」をでっち上げて呼び出そうとした場合、通常はシステムエラーとなります。`ToolHallucinationInterceptor` はこれを防ぎます。

* **異常検知**: ストリーム上で発行されたツール名が、現在システムに登録されているツール群に存在しないことをリアルタイムで検知します。
* **パラメータの再構築**: 存在しないツールへ渡されようとしていた引数ストリームをバッファリングし、最も安全な代替ツール（フォールバックツール）のスキーマに合わせて引数を再マッピングします。
* **透過的な置換**: ユーザーや LLM にエラーを返すことなく、有効なツールへの呼び出しとしてストリームを書き換えて実行を継続します。

## ✨ Key Features

### 🔌 OpenAI API 完全互換

FastAPI ベースのサーバーは、OpenAI の `/v1/chat/completions` エンドポイントと完全に互換性があります。ストリーミング（SSE）やツール呼び出し（Function Calling）のフォーマットも準拠しているため、既存の OpenAI エコシステム（各種 SDK、GUI クライアント、エージェントフレームワーク）から変更なしで Cingulater をバックエンドとして利用可能です。

### ⚙️ 柔軟なモデル構成 (`config.yaml`)

LLM のモデルや接続先はハードコードされておらず、`config.yaml` を書き換えるだけで簡単に変更できます。

* **モデルの変更**: `llm.models` セクションを編集することで、使用するモデル（例: `mlx-community/Qwen3.5-9B-MLX-4bit` など）を自由に指定可能です。
* **エンドポイントの指定**: ローカルの推論サーバーや外部 API など、接続先のエンドポイント URL (`interlocutor_endpoint`) も設定可能です。
* **動的ランチャー**: `launcher_client` と `launcher_tool` を設定することで、Orchestrator 起動時に指定したモデルを MCP 経由で自動的に起動（例：`mlx-launcher` を使用したローカル LLM の起動）させることができます。

### 🛠️ MCP (Model Context Protocol) 統合

`mcp_config.json` を通じて、ローカルプロセスや外部ツールを標準化されたプロトコルで動的に接続・フェッチします。これにより、エージェントの能力（ツールセット）を安全かつ疎結合に拡張できます。

## 🏗️ Architecture

Cingulater は非同期ストリーミングサーバーとして構築されており、以下のコンポーネントで構成されています。

* **Orchestrator**: プロセス全体を統括し、MCP クライアントのライフサイクル管理や LLM クライアントへのリクエストを制御します。
* **Interceptor Pipeline**: リクエストの事前処理と、レスポンスストリームの事後処理を直列に繋ぎます。ログ記録、コンテキスト制限、システムプロンプトの注入、そして上記の自己修復ロジックが含まれます。
* **Workflow Pipeline**: ツールフェッチやアクションの実行など、アクター（LLM）を中心としたタスクの進行を管理します。

## 🚀 Getting Started

### 1. インストールとセットアップ

`uv` および `git-lfs` 等の必要な依存関係を含め、環境を自動構築し、`config.yaml` に基づいてモデルをダウンロードします。

```bash
./bin/setup.sh
```

### 2. サーバーの起動

Orchestrator が初期化され、MCP クライアントの接続と API サーバーがポート `8137` で起動します。

```bash
./bin/cingulater start
```

### 3. インタラクティブ CLI

サーバー起動後、CLI ツールからエンジンと直接対話を行い、エージェントの自律的な動作を確認できます。

```bash
./bin/cingulater chat
```

### 4. クリーンアップ

構築した仮想環境、キャッシュ、データベースなどを安全に削除します。

```bash
./bin/unsetup.sh
```
